Docker & Container Architecture in MyMiniCloud

📷 Add your image here (recommended: 1280×560px)

MyMiniCloud simulates a full-scale cloud platform by running each service as an isolated Docker container. Rather than deploying monolithic software, we decompose the system into 9 specialized services, each communicating over a shared internal Docker network called cloud-net.

Why Containers?

Containers provide process isolation, reproducibility, and portability. Each service has its own filesystem, network interface, and dependency set — just like VMs, but far more lightweight. On a single EC2 machine, we can run all 9 services simultaneously with minimal overhead.

The 9 Services

  • web-server — Nginx serving static HTML/CSS/JS on port 8080
  • app-server — Flask REST API on port 8085 (internal: 8081)
  • db-server — PostgreSQL database on port 3306
  • auth-server — JWT authentication service on port 8081
  • storage-server — MinIO object storage on port 9000/9001
  • dns-server — CoreDNS name resolution on port 1053/UDP
  • monitoring-server — Prometheus metrics scraper on port 9090
  • node-exporter — Host metrics exporter on port 9100
  • logging-server — Grafana visualization dashboard on port 3000
  • proxy-server — Nginx reverse proxy / load balancer on port 80

Internal Networking

All containers are attached to a single bridge network cloud-net. This means services can reach each other by container name — for example, app-server connects to db-server:5432 without knowing the actual IP. Docker's embedded DNS handles hostname resolution automatically.

Starting the Platform

# Build and start all 9 containers
docker compose up --build -d

# Check running services
docker compose ps

# View logs of a specific service
docker compose logs -f app-server

The entire platform boots in under 30 seconds on a t2.small EC2 instance. Each service is independently restartable, upgradeable, and scalable.
