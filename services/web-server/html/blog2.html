Nginx as a Reverse Proxy in MyMiniCloud

📷 Add your image here (recommended: 1280×560px)

In MyMiniCloud, Nginx plays two distinct roles: a web server that delivers static content, and a reverse proxy that acts as the single entry point for all incoming traffic. Understanding this dual role is key to understanding the platform's architecture.

What is a Reverse Proxy?

A reverse proxy sits in front of your backend services. When a user makes a request to port 80, Nginx receives it and decides which internal service should handle it — the user never communicates directly with Flask, PostgreSQL, or MinIO.

User Browser │ ▼ :80 proxy-server (Nginx) │ ├─── /api/* ──► app-server:8081 ├─── /auth/* ──► auth-server:5000 ├─── /storage/* ──► storage-server:9000 └─── /* ──► web-server:80

Benefits in This Setup

  • Single entry point — only port 80 needs to be open to the internet
  • Path-based routing — /api/* goes to Flask, /* goes to Nginx static
  • SSL termination — HTTPS can be configured once at the proxy, not per service
  • Load balancing — easily scale by adding more app-server instances

Key nginx.conf Configuration

upstream app_backend {
    server app-server:8081;
}

server {
    listen 80;

    # Route API calls to Flask
    location /api/ {
        proxy_pass http://app_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # Serve static files via web-server
    location / {
        proxy_pass http://web-server:80;
    }
}

web-server vs proxy-server

Both containers run Nginx but serve different purposes. web-server only knows about static files in /usr/share/nginx/html. proxy-server knows about all the backend services and acts as the traffic controller. This separation makes each service independently replaceable.
